#+TITLE: 配置体系 / Configuration Guide

* 中文版

Brale 的配置分为四层，从全局到单币种逐级细化。所有配置均为 TOML 格式。

** 1) 系统配置：configs/system.toml

全局运行参数：

- 日志：格式、级别、输出路径
- 数据库：PostgreSQL / TimescaleDB DSN
- 执行系统：类型（默认 Freqtrade）、API 地址、认证信息
- 调度：定时决策开关、LLM 调用最小间隔
- Webhook：监听地址、任务队列、回退轮询参数
- 通知：总开关（Telegram / 飞书）

数据库运行要求：

1. 启动时会自动执行数据库 migration
2. 当前基础 migration 会创建 =timescaledb= extension、hypertable 与 retention policy
3. 因此运行库需要使用 TimescaleDB，且数据库用户需要具备相应扩展/策略权限

调度运行要求：

1. =enable_scheduled_decision= 控制定时决策总开关
2. =[scheduler].backend= 支持 =builtin= 与 =river=
3. 当 backend = =river= 时，周期性的 observe / decide / reconcile 由 River periodic jobs 驱动，旧 ticker loops 会被禁用
4. builtin 仍保留为兼容回退路径，但部署配置应与实际运行后端一致

对账配置：

#+begin_src toml
[reconcile]
close_recover_after = "10m"   # 平仓状态回收超时，默认 10 分钟
#+end_src

** 2) 币种索引：configs/symbols-index.toml

决定系统实际运行哪些交易对。每个条目至少包含：

- =symbol= — 交易对名称
- =config= — 指向该币种的配置文件
- =strategy= — 指向该币种的策略文件

** 3) 币种配置：configs/symbols/*.toml

控制单个币种的数据采集方式和 Agent 行为：

- K 线周期（intervals）与数量限制（kline_limit）。=intervals= 的第一个元素（最短周期）同时作为 =decisionInterval= 注入到 Agent prompt 中，告知 LLM 当前决策窗口的时间尺度。
- 启用哪些 Agent（indicator / structure / mechanics）
- 是否要求 OI、funding rate、多空比、恐惧贪婪指数等数据
- 技术指标参数（EMA / RSI / ATR / STC / BB / CHOP / StochRSI / Aroon）
- 共识阈值
- 开仓冷却时间
- 各 Agent 的模型路由与温度

当前 =indicators= 段已暴露的字段包括：

- =ema_fast= / =ema_mid= / =ema_slow=
- =rsi_period= / =atr_period=
- =stc_fast= / =stc_slow= / =skip_stc=
- =bb_period= / =bb_multiplier=
- =chop_period=
- =stoch_rsi_period=
- =aroon_period=
- =last_n= / =pretty=

边界说明：

- =TD Sequential= 当前没有对外暴露参数，仍使用代码内固定计算逻辑。
- =SuperTrend= 不属于 =IndicatorConfig=，它来自结构侧的 =TrendPreset=。
- =kline_limit= 必须覆盖最长指标 warmup；拉长 =stoch_rsi_period=、=aroon_period= 等参数时，配置校验也会同步提高最低要求。

*** 记忆系统配置：[memory]

币种配置中的 =[memory]= 段控制三层记忆系统的行为：

| 参数 | 类型 | 默认值 | 说明 |
|------+------+--------+------|
| =enabled= | bool | false | 记忆系统总开关 |
| =working_memory_size= | int | 5 | 工作记忆保留的最近决策条数 |
| =episodic_enabled= | bool | false | 情景记忆（交易复盘）开关 |
| =episodic_ttl_days= | int | 30 | 情景记忆过期天数 |
| =episodic_max_per_symbol= | int | 50 | 每个币种最多保留的情景记忆条数 |
| =semantic_enabled= | bool | false | 语义记忆（长期规则）开关 |
| =semantic_max_rules= | int | 20 | 每个币种最多保留的语义规则条数 |

记忆系统在决策周期中提供上下文注入：
- *工作记忆*：将最近 N 条决策摘要注入 Agent prompt，提供短期趋势延续
- *情景记忆*：将历史交易复盘注入 Provider prompt，帮助识别重复模式
- *语义记忆*：将长期提炼的规则注入全局上下文，约束 LLM 行为

启用记忆后，每个决策周期额外增加约 500–1K Token。

** 4) 策略配置：configs/strategies/*.toml

控制交易与风控参数：

- =risk_per_trade_pct= — 单笔风险比例
- =max_invest_pct= — 最大资金占用
- =max_leverage= — 最大杠杆
- =entry_mode= — 入场方式（orderbook / atr_offset / market）
- 初始止盈止损策略
- 滑点缓冲
- 持仓收紧（tighten）参数
- sieve 二次风控规则

** Sieve / Grade / Gate Quality 配置

策略配置中的 Gate 与 Sieve 系统控制决策信号的过滤与质量评估。

*** Sieve 规则

Sieve 是二次风控筛选层，在 Gate 通过后对信号做额外校验。配置位于 =configs/strategies/*.toml= 的 =[sieve]= 段：

#+begin_src toml
[sieve]
min_consensus = 2            # 最低共识数：至少 N 个 Provider 同方向才放行
require_indicator = true     # 是否强制要求 indicator Provider 同意
require_structure = false    # 是否强制要求 structure Provider 同意
max_counter_signals = 1      # 允许的最大反向信号数
#+end_src

*** Grade（Agent 置信度映射）

每个 Agent 在决策输出中会附带 =movement_confidence=（0–1 之间的浮点数），表示其对方向判断的置信程度。这个值会映射为 Gate Quality 的权重因子：

- =movement_confidence= 越高，该 Agent 对应的 Provider 布尔信号在 Gate Quality 计算中权重越大
- 低置信度的 Agent 输出会被自动降权，减少噪声对最终判断的影响

*** Gate Quality

Gate Quality 是综合评分，决定一轮决策是否达到执行阈值。计算方式：

1. 每个 Provider 输出一个布尔信号（agree / disagree）
2. 每个信号乘以其对应 Agent 的 =movement_confidence= 作为权重
3. 加权求和后与 =movement_score_threshold= 比较

相关配置位于 =[gate]= 段：

#+begin_src toml
[gate]
movement_score_threshold = 30   # Gate Quality 最低通过分数
consensus_required = 2          # 最低 Provider 共识数
#+end_src

Gate Quality = Σ(Provider_signal × Agent_movement_confidence × 100)。只有当 Gate Quality ≥ =movement_score_threshold= 且共识数 ≥ =consensus_required= 时，决策才会进入 Plan 阶段。

** 添加新币种

推荐使用 =bralectl= 工具一键添加：

#+begin_src bash
# 交互式：从已有币种中选择模板
make add-symbol SYMBOL=XAGUSDT

# 指定模板币种（适合脚本 / CI）
make add-symbol SYMBOL=XAGUSDT TEMPLATE_SYMBOL=SOLUSDT

# 预览模式：只打印将要生成的文件，不实际写入
make add-symbol SYMBOL=XAGUSDT TEMPLATE_SYMBOL=SOLUSDT DRY_RUN=1
#+end_src

该命令会自动完成：

1. 从模板派生 =configs/symbols/XAGUSDT.toml=
2. 从模板派生 =configs/strategies/XAGUSDT.toml=
3. 追加条目到 =configs/symbols-index.toml=
4. 追加 =XAG/USDT:USDT= 到 Freqtrade 的 =pair_whitelist=

添加完成后运行 =make prepare= 重新生成运行时配置，再 =make restart= 使新币种生效。

也可以直接使用 CLI：

#+begin_src bash
make install-bralectl
bralectl add-symbol XAGUSDT --template-symbol SOLUSDT --repo .
#+end_src

* English Version

Brale uses four configuration layers, all in TOML format, from global settings down to per-symbol details.

** 1) System Configuration: configs/system.toml

Global runtime parameters:

- Logging: format, level, output path
- Database: PostgreSQL / TimescaleDB DSN
- Execution: system type (defaults to Freqtrade), API endpoint, authentication
- Scheduling: decision toggle, minimum LLM call interval
- Webhook: listening address, task queue, fallback polling
- Notifications: master switch (Telegram / Feishu)

Database runtime requirements:

1. Startup runs database migrations automatically
2. The base migration creates the =timescaledb= extension, hypertables, and retention policies
3. In practice this means the runtime database must be TimescaleDB, and the DB user needs the required extension/policy privileges

Scheduling runtime requirements:

1. =enable_scheduled_decision= is the master switch for scheduled decisions
2. =[scheduler].backend= supports =builtin= and =river=
3. When backend = =river=, periodic observe / decide / reconcile execution is driven by River jobs and the legacy ticker loops are disabled
4. builtin remains as a compatibility fallback, but deployed configs should match the actual runtime backend

Reconcile config:

#+begin_src toml
[reconcile]
close_recover_after = "10m"   # timeout before recovering close state, defaults to 10 minutes
#+end_src

** 2) Symbol Index: configs/symbols-index.toml

Determines which trading pairs the system actually runs. Each entry requires at minimum:

- =symbol= — the trading pair name
- =config= — path to the symbol configuration file
- =strategy= — path to the strategy configuration file

** 3) Per-Symbol Configuration: configs/symbols/*.toml

Controls data collection and Agent behavior for a single symbol:

- candlestick intervals and kline_limit. The first element of =intervals= (shortest timeframe) is also injected into Agent prompts as =decisionInterval=, informing the LLM of the current decision window's time scale.
- which Agents to enable (indicator / structure / mechanics)
- whether to require OI, funding rate, long-short ratio, fear-greed index, etc.
- technical indicator parameters (EMA / RSI / ATR / STC / BB / CHOP / StochRSI / Aroon)
- consensus thresholds
- entry cooldown
- model routing and temperature for each Agent

The =indicators= section currently exposes:

- =ema_fast= / =ema_mid= / =ema_slow=
- =rsi_period= / =atr_period=
- =stc_fast= / =stc_slow= / =skip_stc=
- =bb_period= / =bb_multiplier=
- =chop_period=
- =stoch_rsi_period=
- =aroon_period=
- =last_n= / =pretty=

Boundary notes:

- =TD Sequential= does not expose tunable config yet and still uses code-level defaults.
- =SuperTrend= is not part of =IndicatorConfig=; it comes from structure-side =TrendPreset= values.
- =kline_limit= must cover the longest indicator warmup. If you increase =stoch_rsi_period=, =aroon_period=, or similar values, validation raises the minimum required history accordingly.

*** Memory System: [memory]

The =[memory]= section in per-symbol config controls the three-layer memory system:

| Parameter | Type | Default | Description |
|-----------+------+---------+-------------|
| =enabled= | bool | false | Master switch for the memory system |
| =working_memory_size= | int | 5 | Number of recent decisions kept in working memory |
| =episodic_enabled= | bool | false | Episodic memory (trade reflections) toggle |
| =episodic_ttl_days= | int | 30 | Episodic memory expiration in days |
| =episodic_max_per_symbol= | int | 50 | Max episodic memories per symbol |
| =semantic_enabled= | bool | false | Semantic memory (long-term rules) toggle |
| =semantic_max_rules= | int | 20 | Max semantic rules per symbol |

Memory provides context injection during decision cycles:
- *Working memory*: injects recent N decision summaries into Agent prompts for short-term trend context
- *Episodic memory*: injects trade reflections into Provider prompts for pattern recognition
- *Semantic memory*: injects long-term rules into global context to constrain LLM behaviour

Enabling memory adds roughly 500–1K tokens per decision cycle.

** 4) Strategy Configuration: configs/strategies/*.toml

Controls trading and risk-management parameters:

- =risk_per_trade_pct= — risk percentage per trade
- =max_invest_pct= — maximum capital usage
- =max_leverage= — maximum leverage
- =entry_mode= — entry method (orderbook / atr_offset / market)
- initial take-profit / stop-loss strategy
- slippage buffer
- in-position tightening parameters
- sieve secondary risk-control rules

** Sieve / Grade / Gate Quality Configuration

The Gate and Sieve systems in strategy configs control decision signal filtering and quality assessment.

*** Sieve Rules

Sieve is a secondary risk-control filter that runs after Gate to apply additional validation. Configured in the =[sieve]= section of =configs/strategies/*.toml=:

#+begin_src toml
[sieve]
min_consensus = 2            # minimum consensus: at least N Providers must agree on direction
require_indicator = true     # require indicator Provider to agree
require_structure = false    # require structure Provider to agree
max_counter_signals = 1      # max allowed counter-directional signals
#+end_src

*** Grade (Agent Confidence Mapping)

Each Agent outputs a =movement_confidence= value (float between 0 and 1) representing its directional conviction. This value maps to the weight factor used in Gate Quality computation:

- Higher =movement_confidence= gives more weight to the corresponding Provider's boolean signal in the Gate Quality calculation
- Low-confidence Agent outputs are automatically down-weighted, reducing noise impact on the final verdict

*** Gate Quality

Gate Quality is the composite score that determines whether a decision round reaches execution threshold. The computation:

1. Each Provider outputs a boolean signal (agree / disagree)
2. Each signal is multiplied by its corresponding Agent's =movement_confidence= as a weight
3. The weighted sum is compared against =movement_score_threshold=

The relevant config lives in the =[gate]= section:

#+begin_src toml
[gate]
movement_score_threshold = 30   # minimum gate quality score to pass
consensus_required = 2          # minimum number of Provider agreements
#+end_src

Gate Quality = Σ(Provider_signal × Agent_movement_confidence × 100). A decision advances to the Plan stage only when Gate Quality ≥ =movement_score_threshold= and consensus count ≥ =consensus_required=.

** Adding New Symbols

Use the =bralectl= tool for one-step setup:

#+begin_src bash
# Interactive: select an existing symbol as template
make add-symbol SYMBOL=XAGUSDT

# Specify template symbol (for scripts / CI)
make add-symbol SYMBOL=XAGUSDT TEMPLATE_SYMBOL=SOLUSDT

# Dry-run: preview files to be generated without writing
make add-symbol SYMBOL=XAGUSDT TEMPLATE_SYMBOL=SOLUSDT DRY_RUN=1
#+end_src

This command automatically:

1. Derives =configs/symbols/XAGUSDT.toml= from the template
2. Derives =configs/strategies/XAGUSDT.toml= from the template
3. Appends an entry to =configs/symbols-index.toml=
4. Appends =XAG/USDT:USDT= to Freqtrade's =pair_whitelist=

After adding, run =make prepare= to regenerate runtime configs, then =make restart= for the new symbol to take effect.

You can also use the CLI directly:

#+begin_src bash
make install-bralectl
bralectl add-symbol XAGUSDT --template-symbol SOLUSDT --repo .
#+end_src
